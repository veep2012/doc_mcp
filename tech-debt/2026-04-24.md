# Tech Debt

Deferred implementation items captured during doc_mcp work.

## TD-20260424-01
- Title: Support reindexing only a selected list of pages
- Context: The current reindex flow is site-scoped and crawls from the configured start URL, which forces a full-site pass even when only a few pages need to be refreshed.
- Location: `crawl_cli.py`, `src/docmcp/tools.py`, `document/requirements.md`
- Impact/Risk: Reindexing large sites is slower than necessary, increases crawl load, and makes targeted refreshes impossible for users who already know the exact pages that changed.
- Why Deferred Now: The immediate task is to record the limitation, not redesign the crawler/API surface or add the supporting tests and storage behavior.
- Proposed Fix: Add a page-selection parameter to the reindex command/tool so callers can supply a list of page URLs or canonical page IDs; update the crawl flow to visit only those pages and keep the index behavior consistent for the selected subset.
- Acceptance Signal: `reindex_site` or its CLI equivalent accepts an explicit page list and reindexes only those pages, with tests proving non-selected pages are left untouched.
- Status: open

## TD-20260424-02
- Title: Auth type variants are not fully tested
- Context: The site configuration sample exposes `auth_type` values such as `email_code`, `magic_link`, and `password_only`, but the runtime auth flow is implemented and validated around the manual `headful` browser path.
- Location: `config/sites.yaml.example`, `src/auth/session.py`, `auth_cli.py`, `README.md`
- Impact/Risk: Users may assume the listed auth types are supported and reliable, but the current implementation does not thoroughly test or guarantee behavior for `email_code` or `password_only`, which can lead to failed logins and repeated session setup.
- Why Deferred Now: The current change is documentation-first; expanding and validating additional auth modes would require implementation work, site-specific test coverage, and likely new flow handling that is outside scope.
- Proposed Fix: Either remove unsupported auth-type guidance from the sample config or implement and test each auth type explicitly with dedicated flows, automated coverage, and clear runtime validation.
- Acceptance Signal: The docs and sample config only advertise auth modes that are verified by tests, or each auth type has a passing automated test path and documented support contract.
- Status: open
