# Technical Debt

Deferred issues discovered during development and audit work.

## TD-20260523-01
- Title: Add repeatable search performance measurements with and without vector search
- Context: The vectorizer and search path were optimized and instrumented, but there is still no documented benchmark that compares keyword-only search against keyword-plus-vector search on the same site data.
- Location: [`src/docmcp/vector_index.py`](/Users/alekseikrutskikh/doc_mcp/src/docmcp/vector_index.py), [`src/docmcp/tools.py`](/Users/alekseikrutskikh/doc_mcp/src/docmcp/tools.py), [`tests/test_tools.py`](/Users/alekseikrutskikh/doc_mcp/tests/test_tools.py), [`tests/test_vector_index.py`](/Users/alekseikrutskikh/doc_mcp/tests/test_vector_index.py), [`documentation/semantic_search_implementation_stages.md`](/Users/alekseikrutskikh/doc_mcp/documentation/semantic_search_implementation_stages.md)
- Impact/Risk: Without a benchmark, it is hard to prove whether vector search improves search quality or latency, and regressions can slip in unnoticed when the vector path changes.
- Why Deferred Now: The current task focused on making vectorization visible and preventing it from hanging; adding a full benchmark harness would expand scope and require deciding the exact dataset, metrics, and pass/fail thresholds.
- Proposed Fix: Add a reproducible benchmark script or test fixture that measures search latency and result quality for keyword-only search and vector-enabled search on the same corpus, then document the expected numbers and how to run the comparison.
- Acceptance Signal: A checked-in benchmark exists, produces comparable metrics for both modes, and the results are referenced from the search documentation.
- Status: open

## TD-20260523-02
- Title: Replace deterministic hash-based embeddings with a real AI/ML embedding model
- Context: The current vectorizer generates embeddings in `_embed_text()` using token hashing and fixed-size vectors instead of a learned model.
- Location: [`src/docmcp/vector_index.py`](/Users/alekseikrutskikh/doc_mcp/src/docmcp/vector_index.py), especially `_embed_text()` and the `embedding_dimensions` configuration path in [`documentation/configuration.md`](/Users/alekseikrutskikh/doc_mcp/documentation/configuration.md)
- Impact/Risk: Hash-based embeddings are deterministic but not semantically meaningful, so vector search quality is limited and may not reflect real retrieval relevance.
- Why Deferred Now: Replacing the embedder requires choosing a provider/model, adding inference dependencies, and defining how embeddings are generated and cached without blocking the current local vector pipeline.
- Proposed Fix: Integrate a real embedding provider or local ML model, make the model configurable, regenerate the vector sidecar from the same crawl data, and update the documentation and tests to reflect the new embedding source.
- Acceptance Signal: The vectorizer uses a documented AI/ML embedding model, the model choice is configurable, and vector search results are demonstrably semantic rather than hash-derived.
- Status: open
