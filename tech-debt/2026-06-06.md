# Technical Debt
Deferred implementation items for 2026-06-06.

## TD-20260606-01
- Title: Parallelize crawl page processing
- Context: While discussing crawl throughput improvements, the current crawler still processes page fetch and parse work in a mostly serial flow.
- Location: `src/docmcp/crawl_cli.py`, `src/docmcp/auth/session.py`, `tests/test_crawl_cli.py`, `documentation/crawling.md`
- Impact/Risk: Crawl throughput stays bounded by single-threaded work, but naïve concurrency could introduce duplicate writes, ordering drift, or session/index race conditions.
- Why Deferred Now: The current work needs to stay focused on correctness and existing behavior; concurrency needs a dedicated design and regression pass.
- Proposed Fix: Introduce bounded parallel work for independent crawl steps while keeping write boundaries deterministic and protecting shared session/index state.
- Acceptance Signal: A crawl run can process multiple pages concurrently without duplicate records, corrupted state, or behavior regressions in the crawler tests.
- Status: open

## TD-20260606-02
- Title: Parallelize vectorization chunk processing
- Context: While considering vectorizer performance improvements, chunk embedding and vector-sidecar writes still run in a single sequential pipeline.
- Location: `src/docmcp/vector_index.py`, `src/docmcp/vectorize_cli.py`, `tests/test_vector_index.py`, `tests/test_vectorize_cli.py`, `documentation/configuration.md`
- Impact/Risk: Vectorization remains slower than necessary on larger sites, and careless concurrency could break deterministic output, chunk ordering, or sqlite-vec write safety.
- Why Deferred Now: This needs its own implementation story because concurrency changes the vectorization pipeline and requires dedicated validation.
- Proposed Fix: Add safe parallel processing for independent chunk embedding work, then serialize or order the final vector writes so output stays reproducible.
- Acceptance Signal: Vectorization can use parallel chunk work while preserving stable results, correct vector sidecar contents, and passing regression tests.
- Status: open

## TD-20260606-03
- Title: Add a skip-existing-pages crawl option
- Context: While discussing crawl and re-crawl performance, the crawler always processes discovered pages even when they already exist in the local index.
- Location: `src/docmcp/crawl_cli.py`, `src/docmcp/index_store.py`, `tests/test_crawl_cli.py`, `documentation/crawling.md`, `documentation/troubleshooting.md`
- Impact/Risk: Re-crawls can waste time and reprocess unchanged pages, but a naive skip mode could miss legitimate content refreshes, redirects, or pages that need reindexing after config changes.
- Why Deferred Now: The skip policy needs a clear freshness rule and explicit operator controls before it can be added without surprising indexing behavior.
- Proposed Fix: Add an opt-in crawl mode that can detect already indexed pages and skip or short-circuit them based on a documented freshness policy, while still allowing full refresh runs when needed.
- Acceptance Signal: The crawler can avoid reprocessing already indexed pages when the skip mode is enabled, without breaking normal full crawls or refresh/rebuild workflows.
- Status: open
